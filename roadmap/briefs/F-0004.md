## Spec

### 1. `next.config.ts` — enable standalone output
Currently `const nextConfig: NextConfig = {};`. Change to:
```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "standalone",
  // Prisma's generated client lives outside node_modules (prisma/generated);
  // ensure Next's standalone file-tracing bundles it into the container.
  outputFileTracingIncludes: {
    "/api/health": ["./prisma/generated/**"],
  },
};

export default nextConfig;
```
`output: "standalone"` makes `next build` emit a self-contained `.next/standalone/server.js`. The `outputFileTracingIncludes` entry is defensive insurance that the Prisma generated client (imported via a relative path from `src/lib/prisma.ts`) is traced into the standalone bundle. `next start` (i.e. `npm run start`) keeps working unchanged after this.

### 2. `src/app/api/health/route.ts` — health endpoint (NEW FILE; create `src/app/api/health/`)
Keep it dependency-free and DB-free (a health check must not depend on the database being seeded, and must not query the DB at build time):
```ts
// Liveness probe for the staging host (Fly.io health checks hit this).
// Intentionally DB-free: it proves the server process is up and serving.
export const dynamic = "force-dynamic";

```
Use the web-standard `Response.json` (no `next/server` import) so the handler is trivially unit-testable in a Node Vitest environment. `force-dynamic` keeps it out of static optimization.

### 4. `tests/deploy-config.test.ts` — config-contents assertions (NEW FILE)
Satisfies acceptance #2 deterministically (config asserted "by a test or the evaluator" — we do both). Read the files from the repo root with `node:fs` and assert the invariants below. The **critical** invariant is consistency: the fly.toml volume mount destination must be a directory prefix of the `DATABASE_URL` SQLite file path, i.e. "the volume mounts at the SQLite path."
```ts
// @vitest-environment node
import { describe, it, expect } from "vitest";
import { readFileSync } from "node:fs";
import { resolve } from "node:path";

const root = resolve(__dirname, "..");
const dockerfile = readFileSync(resolve(root, "Dockerfile"), "utf8");
const flytoml = readFileSync(resolve(root, "fly.toml"), "utf8");
const nextcfg = readFileSync(resolve(root, "next.config.ts"), "utf8");

describe("deploy config", () => {
  it("next.config enables standalone output", () => {
    expect(nextcfg).toMatch(/output:\s*["']standalone["']/);
  });
  it("Dockerfile builds from the standalone output and runs server.js", () => {
    expect(dockerfile).toMatch(/\.next\/standalone/);
    expect(dockerfile).toMatch(/server\.js/);
    expect(dockerfile).toMatch(/\.next\/static/);
    expect(dockerfile).toMatch(/HOSTNAME/);
  });
  it("fly.toml mounts a persistent volume at the SQLite directory", () => {
    const dest = flytoml.match(/destination\s*=\s*["']([^"']+)["']/);
    const dbUrl = flytoml.match(/DATABASE_URL\s*=\s*["']file:([^"']+)["']/);
    expect(dest).not.toBeNull();
    expect(dbUrl).not.toBeNull();
    // The SQLite file must live UNDER the mounted volume directory.
    expect(dbUrl![1].startsWith(dest![1])).toBe(true);
  });
  it("fly.toml health check targets /api/health", () => {
    expect(flytoml).toMatch(/\/api\/health/);
  });
});
```
You may adjust the regexes to match your exact file wording, but **do not weaken the cross-consistency assertion** (DB file path lives under the mount destination) — that is the heart of acceptance #2.

### 5. `Dockerfile` (NEW FILE, repo root) — multi-stage, standalone runtime, seed baked in
Requirements (write clean, commented Dockerfile; exact wording is yours but it MUST satisfy these):
- **Base image:** `node:24-bookworm-slim` for BOTH stages (Debian glibc — `better-sqlite3` prebuilt binaries install cleanly here; avoid Alpine/musl native-module pain). Node 24 matches CI.
- **Builder stage:** `COPY . .` **before** `npm ci` (the `postinstall` runs `prisma generate`, which needs the schema present — copying package files alone first would fail). Then `RUN npm ci`, `RUN npm run build` (emits standalone), `RUN npm run seed` (produces a seeded `prisma/dev.db` to bake in as the volume's first-boot contents).
- **Runner stage:** copy only the standalone artifacts:
  - `COPY --from=builder /app/.next/standalone ./`
  - `COPY --from=builder /app/.next/static ./.next/static`
  - `COPY --from=builder /app/public ./public`
  - `COPY --from=builder /app/prisma/dev.db /app/seed/answerflow.db`  (baked seed db)
- **Runtime env:** `ENV NODE_ENV=production PORT=3000 HOSTNAME=0.0.0.0 DATABASE_URL=file:/data/answerflow.db`. **`HOSTNAME=0.0.0.0` is mandatory** — the standalone server binds localhost otherwise and Fly cannot reach it.
- **`EXPOSE 3000`.**
- **Entrypoint (inline in the Dockerfile `CMD` — do NOT create a separate `.sh` file; a separate entrypoint script is not in authorized_paths and would also be unlinted):** on first boot, copy the baked seed DB onto the empty volume, then exec the server:
  ```dockerfile
  CMD ["sh", "-c", "mkdir -p /data && if [ ! -f /data/answerflow.db ]; then cp /app/seed/answerflow.db /data/answerflow.db; fi && exec node server.js"]
  ```
  This makes the persistent volume hold the SQLite file, gives the operator a populated staging site on first deploy, and needs no Prisma CLI at runtime.

### 8. `docs/DEPLOY.md` (NEW FILE) — operator guide, 8th-grade plain English, click-by-click
Audience: the business operator, non-technical (CLAUDE.md §8). Plain English; explain what each command does and what they'll see; name the small monthly cost honestly. Required sections, in order:
1. **What you're doing** — putting AnswerFlow on the internet so anyone with the link can try it; costs a few dollars a month; you'll do this once.
2. **What you need first** — a Fly.io account (link), a credit card for Fly's billing, and the Fly command-line tool. Give the install command for Windows (`powershell -Command "iwr https://fly.io/install.ps1 -useb | iex"`) and macOS/Linux (`curl -L https://fly.io/install.sh | sh`) and say in plain words what it installs.
3. **Sign in** — `fly auth login` (opens a browser).
4. **Create the app and its storage** — explain `fly launch --no-deploy` will read the included `fly.toml`; then `fly volumes create answerflow_data --size 1 --region iad` creates the 1 GB disk that keeps the data safe between updates. Explain in plain words that this disk is where the questionnaire data lives.
5. **Put it online** — `fly deploy`. Tell them it takes a few minutes and what success looks like.
6. **Open it** — `fly open` (or the URL Fly prints). Mention the health page `…/api/health` shows `{"status":"ok"}` when it's healthy — a quick way to confirm it's up.
7. **What's already inside** — it comes pre-loaded with sample data so it isn't empty when they first look.
8. **Updating later / starting over** — `fly deploy` to push updates; a short note on how to wipe and reseed (destroy + recreate the volume) if they want a clean slate.
9. **If something goes wrong** — `fly logs` shows what's happening; one line that the very first deploy compiles a database driver from source, so it can take a couple extra minutes.
Do not include file paths, stack traces, or jargon in the prose beyond the exact commands they must type. Every command gets a plain-English "what this does / what you'll see."
