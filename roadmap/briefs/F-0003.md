## Gate + CI contracts (facts, verified by reading the code — do not re-derive)
- `verify.sh --e2e` runs the FULL standard gate (typecheck → lint → unit tests → build → …) and then, only if `package.json` defines an `e2e` script: `npx ts-node scripts/seed.ts` (the shim delegates to `npm run seed`, which already works — F-0002) followed by `npm run --silent e2e`. No `e2e` script defined = the gate FAILS by design.
- CI workflow `.github/workflows/e2e.yml` (forbidden surface — you cannot edit it) runs on ubuntu-latest: `npm ci` then `bash scripts/verify.sh --e2e`. **It has NO Playwright browser-install step. Your `e2e` npm script must be self-sufficient** (see §3).
- Unit-test step note: `npm test` (Vitest) runs BEFORE e2e inside the gate; your changes must not break it (see §5 for the glob collision).

## Stack facts (live-verified 2026-06-11 — override memory)
- Current stable `@playwright/test`: **1.60.0** (2026-05-11). Bundled Chromium 148. Install as devDependency.
- `@playwright/test`/`playwright`/`playwright-core` have **no npm install scripts** — no `npm approve-scripts` entries needed (do not add any). The only scripted transitive dep is `fsevents`, darwin-only, never installs on Windows/Linux.
- `webServer.port` is **deprecated** — use `url`. Documented convention: `reuseExistingServer: !process.env.CI`. Default webServer timeout is 60s; Next dev first-compile is lazy, so set 120s (Vercel's own Playwright example does exactly this).
- **Project-dependencies setup is the doc-recommended pattern** (a `setup` project other projects depend on), preferred over `globalSetup` — use it for the seed step.
- Docs' CI browser install: `npx playwright install --with-deps chromium`. `--only-shell` additionally skips full Chromium and keeps the headless shell (~113 MiB instead of ~288 MiB) — we run headless everywhere, use it. On Windows `--with-deps` is harmless (no apt step).
- Playwright transpiles its own test files (Babel) and does no type-checking; `tsc --noEmit` already covers `tests/**` via tsconfig include — that's fine and wanted.

## Spec

### 1. Dependency
Add `@playwright/test` ^1.60.0 as devDependency. **After any dependency change, a full local `npm ci` must succeed before you finish** (standing rule from F-0001).

### 2. `playwright.config.ts` (repo root)
- `testDir: "tests/e2e"`.
- Two projects: `setup` (testMatch for `tests/e2e/global.setup.ts`) and `chromium` (Desktop Chrome device, `dependencies: ["setup"]`).
- `webServer`: `{ command: "npm run dev -- -p 3100", url: "http://localhost:3100", reuseExistingServer: !process.env.CI, timeout: 120_000 }`. Port 3100 on purpose — never collide with an operator's dev server on 3000.
- `use: { baseURL: "http://localhost:3100", trace: "on-first-retry" }`; `retries: process.env.CI ? 1 : 0`; `workers: process.env.CI ? 1 : undefined`.
- `reporter: [["list"], ["html", { open: "never" }]]` — never auto-open a report server (agents run this unattended).
- `forbidOnly: !!process.env.CI`.

### 3. npm script (package.json)
- `"e2e": "playwright install --with-deps --only-shell chromium && playwright test"` — self-sufficient: first run downloads the headless shell (CI has no install step; cached runs are a fast no-op), then runs the suite. Exit code must propagate (the `&&` chain does).
- Do not touch any existing script.

### 4. Seed-before-tests — `tests/e2e/global.setup.ts`
The `setup` project runs the real contract: `execSync("npm run seed", { stdio: "inherit" })` from the repo root. Deterministic baseline state for every E2E run, regardless of caller. (The gate also seeds via the shim before `npm run e2e` — double-seeding is safe because the seed is an idempotent reset; do NOT invoke `scripts/seed.ts` yourself, it's forbidden surface.)

### 5. Keep Vitest and Playwright out of each other's globs
Their default test-match globs fully overlap (`**/*.{test,spec}.*`). Playwright is scoped by `testDir: tests/e2e`. For Vitest, edit `vitest.config.ts` (authorized): `exclude: [...configDefaults.exclude, "tests/e2e/**"]` — you MUST spread `configDefaults.exclude` (from `vitest/config`), because setting `exclude` replaces the defaults. Touch nothing else in that file. Existing unit tests (`tests/smoke.test.tsx`, `tests/seed.test.ts`) must keep passing.

### 6. Dashboard renders seeded data — minimal product change (src/)
The smoke spec must see real seeded content, so the dashboard page (`src/app/page.tsx`) must list projects from the database:
- Make it an async server component reading via the shared client (`src/lib/prisma.ts`): fetch projects (id, name, createdAt and a question count via `_count` or equivalent).
- **`export const dynamic = "force-dynamic"` on this page — mandatory.** Why: `next build` runs inside plain `verify.sh` where no database is guaranteed; force-dynamic keeps the query out of build-time prerender so the build never depends on a DB file existing.
- Render each project's name (the seeded one is "Acme Corp Security Questionnaire") and an explicit, plain-English empty state when there are no projects (the page must never crash on an empty/missing-table DB — wrap the query so a missing db yields the empty state rather than a 500; F-0007 builds on this page).
- Keep the existing shell (Navbar, dark theme, AnswerFlow branding) intact — the existing smoke render test must keep passing. If that Vitest render test breaks because the page is now async, adapt the TEST to render the resolved component (await the async component or test the page's child pieces) — without deleting or weakening any assertion; explain exactly what you changed in notes.md.
- If the build/dev server errors on bundling native modules (better-sqlite3), Next's default `serverExternalPackages` list already includes Prisma and better-sqlite3 — only if you still hit it, add the minimal `serverExternalPackages` entry to `next.config.ts` (authorized) and document it in notes.md.
