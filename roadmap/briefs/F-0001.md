## 5. THE critical landmine: one root tsconfig.json serves three masters

Current root `tsconfig.json`: `module: NodeNext`, `moduleResolution: NodeNext`, `strict`, `esModuleInterop`, `skipLibCheck`, `noUnusedLocals`, `types: ["node"]`, `include: ["scripts/**/*.ts"]`.

After your change, the SAME file must serve:
1. **Next.js** (dev/build) — wants `jsx: preserve`, bundler-style resolution, `include` covering `src/` + `next-env.d.ts`. Next may auto-edit tsconfig.json on first build; let it, then reconcile by hand.
2. **`tsc --noEmit`** (the typecheck gate) — must typecheck `src/**`, `tests/**`, AND keep covering `scripts/**` (do not silently drop engine coverage; if Next-required options genuinely conflict with `scripts/` code, fix the smallest thing and flag it in your report).
3. **ts-node at runtime** (`npx ts-node scripts/update-state.ts` etc., invoked by verify.sh and hooks; ts-node resolves the root tsconfig). Use ts-node's tsconfig override block so engine scripts keep NodeNext semantics regardless of what Next wants:

```jsonc
{
  "compilerOptions": { /* Next-compatible options */ },
  "include": ["next-env.d.ts", "src/**/*.ts", "src/**/*.tsx", "tests/**/*.ts", "tests/**/*.tsx", "scripts/**/*.ts"],
  "exclude": ["node_modules", ".next"],
  "ts-node": {
    "compilerOptions": {
      "module": "NodeNext",
      "moduleResolution": "NodeNext",
      "jsx": "react-jsx"
    }
  }
}
```

You are NOT authorized to touch `scripts/**` — you cannot add `scripts/tsconfig.json`. The override block is the mechanism. After ANY tsconfig change, immediately smoke-test: `npx ts-node scripts/update-state.ts --validate` must still exit 0.
